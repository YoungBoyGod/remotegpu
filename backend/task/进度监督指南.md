# RemoteGPU 进度监督指南

## 文档说明

本文档为项目经理提供详细的监督指南，帮助：
1. 及时发现问题和风险
2. 确保项目按计划推进
3. 保证代码质量
4. 促进团队协作

---

## 监督原则

### 1. 信任但验证
- 信任团队成员的专业能力
- 通过客观指标验证进度
- 不过度干预技术细节

### 2. 预防为主
- 每日检查，及早发现问题
- 关注关键路径和瓶颈
- 主动识别风险

### 3. 数据驱动
- 基于客观数据做决策
- 跟踪关键指标
- 避免主观判断

## 风险识别和处理

### 风险等级定义

**🔴 高风险（立即处理）**:
- 关键路径任务延期>4小时
- 核心功能无法实现
- 团队成员无法工作

**🟡 中风险（密切关注）**:
- 非关键路径任务延期
- 技术难题需要研究
- 依赖关系不明确

**🟢 低风险（正常跟踪）**:
- 小的技术问题
- 文档不完善
- 代码风格问题

---

### 风险处理流程

#### 1. 识别风险
```
每日站会识别
每日下午检查识别
团队成员主动报告
```

#### 2. 评估风险
```
影响范围: 影响多少人？
影响时间: 延期多久？
影响功能: 核心功能还是辅助功能？
```

#### 3. 制定对策
```
🔴 高风险:
- 立即召集相关人员讨论
- 调整资源分配
- 必要时调整计划

🟡 中风险:
- 当天内制定解决方案
- 安排技术支持
- 跟踪解决进度

🟢 低风险:
- 记录问题
- 正常跟踪
- 定期回顾
```

#### 4. 跟踪执行
```
- 每日检查风险状态
- 更新风险清单
- 验证解决效果
```

## 常见问题处理

### 问题1: 任务延期

**识别标志**:
- 每日站会报告进度落后
- Git提交记录少
- 测试未完成

**处理步骤**:
1. 了解延期原因（技术难题？需求不清？其他干扰？）
2. 评估影响范围（是否影响关键路径？）
3. 制定对策：
   - 技术难题：安排技术支持或结对编程
   - 需求不清：立即澄清需求
   - 其他干扰：消除干扰因素
4. 必要时调整计划或重新分配任务

---

### 问题2: 依赖阻塞

**识别标志**:
- 团队成员报告等待其他人
- 任务状态长时间为"等待依赖"

**处理步骤**:
1. 确认依赖关系是否必要
2. 评估被依赖任务的进度
3. 制定对策：
   - 使用mock数据解除依赖
   - 调整开发顺序
   - 加速被依赖任务
4. 更新依赖关系文档

---

### 问题3: 代码质量问题

**识别标志**:
- 测试覆盖率低
- 代码审查不通过
- 频繁出现bug

**处理步骤**:
1. 代码审查，找出问题根源
2. 要求重构或修复
3. 加强代码审查
4. 必要时进行技术培训

---

### 问题4: 团队协作问题

**识别标志**:
- 通知不及时
- 沟通不畅
- 重复工作

**处理步骤**:
1. 召开团队会议，明确协作规范
2. 强化通知机制
3. 更新协作文档
4. 定期回顾协作效果

**每个PR必须检查**:

- [ ] 单元测试完整

- [ ] 表结构与SQL文件一致
